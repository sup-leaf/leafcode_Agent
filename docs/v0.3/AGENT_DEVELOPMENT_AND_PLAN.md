# LeafCode Agent 开发指南与 v0.3 计划

> 文档性质：本文件记录 v0.3 的设计背景、目标架构与取舍，其中“建议”“下一步”等表述不一定代表当前实现状态。当前实施状态请以 [IMPLEMENTATION_PLAN.md](IMPLEMENTATION_PLAN.md)、[ARCHITECTURE.md](ARCHITECTURE.md) 和 [TEST_CASES.md](TEST_CASES.md) 为准。
>
> 2026-07-16 实现摘要：稳定元素 ID、动作验证、结构化运行时事件、Yes/No 风险确认、单工作线程 Playwright 调用和 16 条离线回归已完成；`operate` 保留浏览器会话，`inform` 在任务结束后关闭浏览器；真实网站验收尚未完成。

> 项目定位：LeafCode 是一个可从命令行唤起的、由大语言模型驱动的浏览器 Agent。它通过 Playwright 观察和操作网页，并在终端中以接近 OpenCode 的方式展示任务过程。
>
> 本文最初将已实现阶段定义为 **v0.2**、下一目标定义为 **v0.3**；项目现已进入 v0.3 开发阶段。已有文档中以 TUI 重构阶段命名的 v1–v5 不作为产品版本号。

## 第一章：从零开发 Agent 的普遍流程

Agent 不是“给模型加一个提示词”。一个能稳定完成任务的 Agent，应依次建立目标、能力、循环、可靠性和产品体验五层能力。

### 1. 明确问题与边界

先写清楚 Agent 为谁服务、解决什么任务、绝不做什么。否则工具、提示词和 UI 会不断膨胀而没有可验证的目标。

以浏览器 Agent 为例，应先确定：

- 目标用户：需要检索、阅读、导航网页的个人用户。
- 首批任务：信息检索、网页阅读、打开指定站点、填写低风险表单。
- 非目标：支付、发帖、删除数据、绕过验证码、未经确认地提交重要操作。
- 成功标准：任务完成率、单任务平均步骤数、失败后能否给出可理解的原因。

产物：一页任务范围说明和 10–20 条代表性任务（benchmark）。

### 2. 把自然语言目标拆成可执行工作流

用户说的是目标，例如“查一下北京天气并总结”；浏览器只能执行原子动作。中间需要计划层负责把目标变为可检查的步骤。

```text
用户目标
  → 意图识别（检索 / 阅读 / 操作）
  → 任务计划（查什么、去哪里、何时完成）
  → 原子动作（navigate / click / type / scroll）
  → 验证结果（是否真的到了目标页、是否提取到信息）
  → 汇总或请求确认
```

计划不应只是展示给用户的文本；每一步都应有预期结果和失败回退方案。

### 3. 设计工具层

工具是 Agent 唯一能改变外部世界的接口，因此必须小、明确、可验证。

浏览器 Agent 的基础工具：

| 类别 | 最小工具 | 返回值要求 |
| --- | --- | --- |
| 导航 | `navigate(url)`、`back()` | 当前 URL、标题、是否成功 |
| 感知 | `observe()`、`screenshot()` | 页面摘要、稳定元素 ID、可见文本 |
| 交互 | `click(element_id)`、`type(element_id, text)`、`press(key)`、`scroll()` | 动作是否成功、页面变化 |
| 信息 | `extract(selector_or_goal)` | 结构化内容和来源 URL |
| 生命周期 | `new_tab()`、`switch_tab()`、`finish()` | 当前会话状态 |

原则：不要让模型“猜 CSS 选择器”或依赖每次都会变的元素序号。感知工具应为当前页面生成稳定的元素 ID，操作工具只接受该 ID。

### 4. 建立观察—决策—执行—验证循环

这是 Agent 的核心运行时，而不是一次性调用模型。

```text
Observe → Decide → Act → Verify
  ↑                     │
  └────失败时重试/改计划──┘
```

- Observe：获取 URL、标题、可访问性树/可交互元素、必要的截图和关键文本。
- Decide：模型返回结构化动作；用 schema 校验动作与参数。
- Act：执行一个原子工具调用。
- Verify：判断页面是否发生预期变化；失败时将错误反馈给模型。
- Stop：满足完成条件、达到预算、用户取消，或需要人工确认。

为循环设置硬边界：最大步数、最大时间、最大模型调用次数、重复动作阈值和允许访问的域名。

### 5. 提示词、模型与记忆

提示词应包含角色、工具契约、完成定义、失败处理和安全规则。模型输出必须是 JSON 或工具调用，而不是靠正则从自然语言中“猜”操作。

记忆分三类：

- 任务记忆：当前目标、计划、已完成步骤、失败原因。
- 会话记忆：当前浏览器上下文、标签页、登录状态。
- 长期记忆：用户偏好、常用网站；必须可查看、可删除，且默认最小化保存。

### 6. 可靠性、评测与安全

先用固定任务集评测，再增加能力。每条任务需要记录：输入、步骤、页面状态摘要、工具结果、最终答案、失败类型和耗时。

浏览器 Agent 的最低安全线：

- 输入密码、发送消息、提交表单、上传文件、下载、支付或删除前，要求用户确认。
- 页面内容只是不可信数据，不能覆盖系统规则或诱导泄露密钥。
- 凭证只从本地安全配置读取，禁止显示在日志和模型上下文中。
- 对允许域名、下载路径、文件上传设置显式策略。

### 7. 设计人机协作体验

UI 的任务是让用户随时知道 Agent 在做什么、能否打断、下一步会有什么影响。

最重要的信息层级依次是：任务状态、当前计划、关键动作、需要用户决定的事项、可折叠的调试细节。先保证可控与可解释，再做视觉装饰。

### 8. 迭代顺序

推荐顺序：

1. 固定一类任务，跑通最小闭环。
2. 让元素定位和动作验证稳定。
3. 建立日志与测试集，修复真实失败案例。
4. 加入会话、确认机制和恢复能力。
5. 最后再扩展站点、插件、多 Agent 与复杂 UI。

## 第二章：LeafCode v0.2 现状与 v0.3 设计

### 2.1 v0.2 已有基础

LeafCode 已具备可演示的最小闭环：DeepSeek 负责规划和决策，Playwright 负责浏览器动作，Textual 提供命令行 TUI；支持检索型 `inform` 和保留浏览器窗口的 `operate` 模式。

当前问题不在于缺少更多动作，而在于执行底座还不稳定：

- 搜索结果展示时会过滤部分元素，但展示索引与点击索引可能不一致。
- 每个任务都会新建浏览器；`operate` 虽能保留窗口，但下一轮 Agent 不能继续接管同一会话。
- 页面状态主要依赖文本和元素序号，缺少稳定元素 ID、页面变化验证和标签页管理。
- 凭证读取函数已经存在，但尚未形成安全、可确认的登录工作流。
- `requirements.txt` 未声明 TUI 所需的 `textual`；启动脚本也使用了本机绝对路径。
- 现有 UI 已能输入、滚动和中断，但侧栏、命令面板和任务状态仍是占位实现。

### 2.2 v0.3 的版本目标

**主题：可靠的单会话浏览器 Agent + 清晰的 OpenCode 风格终端体验。**

v0.3 不追求多 Agent、插件或全站登录自动化。验收重点是：同一浏览器会话内能连续完成任务，用户能看懂、能中断、能确认高风险操作，常见检索任务可稳定复现。

建议验收任务：

1. 搜索一个主题，打开两篇结果，输出带来源链接的总结。
2. 打开指定网页，继续下一条指令在同一网页中操作。
3. 浏览页面遇到失败后，展示失败原因并改用一次不同策略。
4. 触发“提交表单”类动作时先暂停，等待用户确认。

### 2.3 v0.3 功能设计（第一优先级）

| 优先级 | 模块 | v0.3 功能 | 验收标准 |
| --- | --- | --- | --- |
| P0 | 稳定元素引用 | `observe()` 生成 `element_id`，展示与点击共用同一份快照 | 不再出现“显示 A、点击 B” |
| P0 | 动作验证 | 每次动作返回 URL/标题/关键 DOM 变化；失败给出明确错误 | 无页面变化时不静默进入下一步 |
| P0 | 浏览器会话 | `BrowserSession` 持有 Playwright、browser、context、page；支持跨任务复用 | 第二条任务能继续操作同一页 |
| P0 | 安全确认 | 表单提交、登录、下载、外发消息等动作进入 `awaiting_confirmation` | 未确认不得执行高风险动作 |
| P1 | 任务状态机 | `planning → running → awaiting_confirmation → completed/failed/cancelled` | TUI 与运行时状态一致 |
| P1 | 计划与恢复 | 保存当前目标、步骤、已执行工具和错误；支持“继续/重试” | 失败后能从当前页面继续 |
| P1 | 信息结果 | 总结附来源 URL；区分事实、推断与失败 | 用户可追溯结论来源 |
| P1 | 配置 | 依赖完整声明；启动脚本使用项目相对路径；配置模型和浏览器参数 | 新环境按 README 可启动 |
| P2 | 登录支持 | 站点适配器 + 经过确认的凭证填充；不处理验证码绕过 | 可完成一个指定站点的登录演示 |

建议的内部结构：

```text
agent.py
  ├─ AgentRuntime        # 状态机、循环、预算、日志
  ├─ Planner             # 意图、计划、完成条件
  ├─ BrowserSession      # 浏览器与跨任务页面会话
  ├─ BrowserObserver     # 生成页面快照和 element_id
  ├─ BrowserActions      # 只接受 element_id 的原子操作
  ├─ SafetyPolicy        # 风险动作识别与确认
  └─ TaskStore           # 任务记录、恢复和可选持久化

agent_tui_v4.py（建议 v0.3 改名为 tui.py）
  ├─ TaskView
  ├─ PlanView
  ├─ ActivityFeed
  ├─ Composer
  └─ CommandPalette
```

### 2.4 v0.3 UI 设计：借鉴 OpenCode，而非复制

视觉目标是“冷静、紧凑、可读、信息密度高”。建议使用深色中性色背景，以蓝色表示可操作/运行中、绿色表示已完成、黄色表示需要确认、红色只用于错误；不要让 Emoji 成为唯一状态表达。

布局建议：

```text
┌ LeafCode · 当前任务 · 模式 · 浏览器状态 ───────────────────────────────┐
│  左栏（可收起）        │ 主区域：任务时间线 / 对话                  │
│  · 当前会话            │  用户目标                                  │
│  · 最近任务            │  Plan  1. 搜索  2. 阅读  3. 总结           │
│  · 浏览器标签页        │  ● 正在阅读：页面标题 / 域名               │
│                         │    可折叠：感知快照、工具参数、调试日志     │
├────────────────────────┴─────────────────────────────────────────────┤
│ [ 输入任务或 / 命令…                                  ]  Inform/Operate │
│ Enter 发送 · Shift+Enter 换行 · Esc 停止 · Ctrl+P 命令面板             │
└───────────────────────────────────────────────────────────────────────┘
```

v0.3 UI 功能清单：

- 顶栏：任务状态、模式、当前域名、耗时、步骤预算。
- 左栏：当前会话、最近任务、当前标签页；默认可收起，避免挤占窄终端。
- 主区：按“计划 / 动作 / 结果 / 错误”显示时间线；原始页面快照和模型思考默认折叠。
- 输入框：优先实现稳定的多行输入；`Enter` 发送、`Shift+Enter` 换行（若 Textual 版本限制，则保留单行并明确提示）。
- 确认卡片：用明确按钮或命令确认 `y/n`，展示将要执行的动作、目标网站和影响。
- 命令面板：`/new`、`/clear`、`/mode`、`/tabs`、`/stop`、`/retry`、`/help`；`Ctrl+P` 搜索命令。
- 无障碍与终端兼容：不依赖颜色区分状态；窄屏自动隐藏侧栏。

### 2.5 实施节奏

#### 里程碑 A：执行底座（P0）

1. 为页面快照定义数据结构：URL、标题、文本摘要、元素列表、截图路径（可选）。
2. 用稳定 `element_id` 替换现有可见元素索引。
3. 抽出 `BrowserSession`，让浏览器跨任务复用，并提供显式关闭命令。
4. 为所有工具统一返回 `ToolResult(success, message, before, after)`。
5. 增加重复动作、超时、无变化检测和结构化日志。

完成标准：四条验收任务中的前两条可稳定完成。

#### 里程碑 B：安全与可恢复性（P1）

1. 引入任务状态机及用户取消逻辑。
2. 为风险动作接入确认卡片和策略层。
3. 记录任务事件，支持重试和从当前页面继续。
4. 总结输出携带来源链接。

完成标准：第三、四条验收任务通过，失败不会丢失当前页面。

#### 里程碑 C：TUI 重构与美化（P1）

1. 按上述三栏布局重构 Textual Widget 树和 CSS token。
2. 实现顶栏、计划卡片、活动时间线、折叠调试详情与确认卡片。
3. 完成多行输入、命令面板、窄屏降级和快捷键帮助。
4. 删除未实现的伪功能或明确标注为“即将推出”。

完成标准：用户不看日志也能回答“它在做什么、做到哪一步、能否安全停止”。

### 2.6 暂不纳入 v0.3

- 多 Agent 协作和插件系统。
- 自动绕过验证码、反爬或复杂网站风控。
- 支付、发布、删除等不可逆动作自动化。
- 大规模长期记忆和云端同步。
- 视觉识别优先级高于 DOM 的纯视觉 Agent。

这些功能应在单 Agent 会话、可靠性和安全模型稳定后进入 v0.4+。

### 2.7 v0.3 Definition of Done

- `pip install -r requirements.txt` 后可启动；启动命令不依赖个人绝对路径。
- 有至少 10 条自动或半自动回归任务，并记录通过率。
- 同一会话可连续接收至少两条浏览器任务。
- 任一风险操作都要用户确认，日志中不出现密钥或密码。
- 终端界面提供清晰任务状态、计划、活动记录、取消和重试入口。
- README 说明安装、配置、运行、快捷键、限制和已知风险。
