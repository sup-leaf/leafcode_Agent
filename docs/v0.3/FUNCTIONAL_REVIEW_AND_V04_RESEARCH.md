# v0.3 功能审查与 v0.4 研究候选

**审查日期：** 2026-07-16  
**结论：** v0.3 已完成本地可演示的浏览器 Agent 闭环，但仍有阻断发布质量的功能与验收缺口。因此本轮**不创建正式的 docs/v0.4 文件夹，也不将 v0.3 标记为完成**。本文件记录补齐 v0.3 的发布门槛，并基于公开资料提炼 v0.4 的候选需求。

## 1. 审查范围与依据

本次审查覆盖：

- 运行时、浏览器、安全策略、日志和 Textual TUI 的当前实现；
- 16 条离线回归测试；
- v0.3 实施计划和 T01–T10 验收任务；
- Browser Use、Playwright、LangGraph/LangChain 与 OpenCode 的公开一手文档。

“已实现”表示本地源码和离线测试能证明；“待验收”表示尚未在真实网站、异常环境或重启场景中获得证据，不能因为存在代码就宣称完成。

## 2. v0.3 当前能力基线

| 能力 | 当前状态 | 本地证据 |
| --- | --- | --- |
| 页面感知 | 已实现 | BrowserTools 为每次观察生成 PageSnapshot、元素 role/text 和临时 element_id。 |

## 3. v0.3 仍存在的功能缺口

### 3.1 发布阻断项（必须先补齐或明确降级）

| 编号 | 缺口 | 现状与风险 | 建议验收 |
| --- | --- | --- | --- |
| G-01 | 真实网站端到端验收缺失 | T01–T10 全部仍为“待测”；本地 HTML 不能覆盖搜索页动态渲染、弹窗、站点加载和真实导航。 | 使用测试账户或公开页面执行至少 10 项任务；记录成功率、步骤数、耗时、失败类别、日志路径。 |
| G-02 | 重启后无法恢复任务 | TaskContext、pending_confirmation 与 Session 只存在内存；进程关闭或崩溃后用户无法继续。 | 在确认等待、动作失败和正常运行三种状态下重启进程，能从持久 checkpoint 恢复或给出可解释的不可恢复提示。 |
| G-03 | 模型输出契约仍靠正则提取 | _extract_json() 使用大括号正则；嵌套、多个 JSON 块、Markdown 代码块和非 JSON 可能误解析。 | 使用严格 schema 校验；非法响应具有分类错误、有限重试与最终可操作提示。 |
| G-04 | 敏感信息边界不完整 | 日志会脱敏已知 key，但页面正文和元素 value 可进入模型上下文；没有字段级最小化、域名策略或提示注入检测。 | 对 password、token、身份证、邮箱等页面字段进行脱敏/排除；加入恶意页面提示注入与日志泄露回归。 |
| G-05 | 异常演练缺失 | 无 API Key、浏览器安装失败、网络超时、窄终端和用户中断未形成完整的自动或人工验收记录。 | 为每类异常建立可重复用例与用户可读错误标准。 |

### 3.2 可靠性和可用性缺口（不阻断本地闭环，但限制真实使用）

| 编号 | 缺口 | 当前实现限制 | v0.4 方向 |
| --- | --- | --- | --- |
| G-06 | 页面定位仅覆盖普通 DOM | 仅扫描有限 selector，未覆盖跨域 iframe、Shadow DOM、复杂菜单或非标准组件。 | 引入可访问性树、frame 作用域、元素语义优先级与站点适配扩展点。 |
| G-07 | 验证维度单一 | 主要比较页面 fingerprint；输入成功、下载完成、列表增减等业务成功条件无法精确描述。 | 为动作定义可选 postcondition，例如 URL、标题、元素文本、输入值、下载文件。 |
| G-08 | 重试策略薄弱 | 重复动作只会停止；没有分类退避、模型降级、备用模型或换策略提示。 | 定义 error taxonomy、有限重试、指数退避和 fallback model。 |
| G-09 | 计划是静态文本 | 计划只用于展示，执行中没有任务图、步骤完成状态、动态改计划或子目标。 | 建立可持久化的计划节点、执行证据和“重新规划原因”。 |
| G-10 | 性能不可观测 | 日志没有统一记录模型耗时、页面耗时、token、动作等待、成功率等指标。 | 为每一步建立 trace/span 与性能仪表盘。 |
| G-11 | TUI 信息架构未收尾 | 没有最近任务、真实标签页列表、截图/trace 入口、多行编辑和可编辑确认参数。 | 建立会话工作台、详情抽屉、确认差异视图与人工接管入口。 |

## 4. 与同类项目的横向比较

| 维度 | LeafCode v0.3 | Browser Use 的公开能力 | 可借鉴但不照搬的结论 |
| --- | --- | --- | --- |
| 会话连续性 | operate 保留当前进程内页面，inform 终态关闭。 | Follow-up tasks 通过 session_id 复用同一浏览器，浏览器状态可延续；云端还提供 keepAlive 配置。 | 把“浏览器存活”和“任务上下文存活”拆开；引入显式 session_id、会话 TTL 与关闭策略。 |
| 动作批量化 | 每轮模型只产出一个动作。 | 提供 max_actions_per_step，在页面变化前可执行多个动作。 | 只允许同页、无外部影响的安全动作批量执行；导航、提交、下载仍单步确认。 |
| 上下文与成本 | 每轮发送正文摘要和元素列表，固定最大 10 步。 | 提供 max_history_items、单独的页面提取模型、flash_mode 与 fallback LLM。 | 需要上下文压缩、DOM 差分、快模型提取与强模型决策的分层路由。 |
| 可靠性 | 指纹变化验证与重复动作熔断。 | 提供 max_failures、备用模型、生命周期 hooks、历史对象。 | 以错误分类、失败预算、trace 和策略切换补足“停止后为什么停止”。 |
| 安全与权限 | 关键词风险判断，Yes/No。 | 支持 allowed/prohibited domains、profile 和敏感数据配置。 | 建立域名策略、字段级敏感数据策略、细粒度确认与可编辑参数。 |
| 人工确认 | approve/reject 二选一。 | LangGraph HITL 模式通常支持 approve、edit、reject、respond，并将状态持久化。 | 增加“编辑后执行”和“拒绝并附反馈”，确认卡片显示动作、域名、参数差异和风险原因。 |
| 可观测性 | JSONL 事件日志；调试面板显示最新参数。 | Browser Use history 可查询 action、URL、错误、耗时和结构化结果；可接入 OTel。 | 用任务 trace 统一串接模型、页面、工具和人工决策。 |
| TUI 风格 | 单时间线 + 侧栏摘要 + 命令面板。 | OpenCode 有会话持久化、不同角色/权限/模型与颜色配置。 | 采用“会话列表 + 计划/时间线 + 可折叠详情 + 权限确认”四层，而不是堆更多颜色和文本。 |

### 4.1 可参考的公开资料

- Browser Use 的 [follow-up tasks 文档](https://docs.browser-use.com/cloud/agent/follow-up-tasks) 说明了浏览器状态与 Agent 上下文可以分离：新任务复用浏览器状态，但不必携带全部推理上下文。
- Browser Use 的 [全部 Agent 参数](https://docs.browser-use.com/open-source/customize/agent/all-parameters) 提供了历史窗口、批量动作、失败上限、备用模型与单独页面提取模型等性能控制点。
- Browser Use 的 [Fast Agent 示例](https://docs.browser-use.com/open-source/examples/templates/fast-agent) 采用更快模型、减少等待、批量动作与关闭显式思考来降低时延；这些优化只能用于低风险、可验证路径。
- Browser Use 的 [输出格式文档](https://docs.browser-use.com/open-source/customize/agent/output-format) 把动作、URL、错误、模型输出和耗时组织为可查询历史，适合作为 LeafCode trace 的参考。
- Playwright 的 [actionability 文档](https://playwright.dev/docs/actionability) 说明 click 会等待元素可见、稳定、可接收事件、可用；v0.4 不应以 force click 绕过这些保障。
- LangChain 的 [Human-in-the-loop 文档](https://docs.langchain.com/oss/python/langchain/human-in-the-loop) 将暂停状态持久化，并支持 approve、edit、reject、respond 四类决策，适合作为确认机制的产品参考。
- OpenCode 的 [Agents 文档](https://opencode.ai/docs/agents/) 展示按角色配置模型、权限、可见性和颜色的方式；LeafCode 可借其信息架构，而无需引入多 Agent。

## 5. v0.4 的设计主题

建议主题为：

> **可靠、可恢复、可度量的浏览器 Agent 工作台**

v0.4 不应先追求更多站点和更多“看起来聪明”的自动化。它应先让每一次任务有稳定 ID、持久状态、可验证证据、性能数据和明确的人机边界。

### 5.1 目标执行架构

~~~text
TUI / CLI
  │  创建、恢复、停止、确认、人工接管
  ▼
TaskService
  ├─ RunStore                 # 任务、计划、检查点、确认、结果
  ├─ Orchestrator             # observe → decide → policy → act → verify
  ├─ ModelRouter              # 计划/提取/决策模型、schema、fallback
  ├─ PolicyEngine             # 域名、敏感字段、权限、确认规则
  ├─ TraceService             # step/span、截图、动作证据、指标
  └─ BrowserSessionManager    # session_id、TTL、标签页、profile、资源回收
        │
        ▼
BrowserAdapter
  ├─ Accessibility/DOM observer
  ├─ Frame / tab / download adapter
  ├─ Action executor
  └─ Postcondition verifier
~~~

设计约束：

1. TaskRun、BrowserSession、Plan 和 Trace 是不同对象，不能再由一个 BrowserAgent 内存实例隐式承担。
2. 每个可产生外部影响的动作都要有 action_id、风险原因、原始参数、用户决定和验证证据。
3. UI 只能调用 TaskService 的命令；它不持有 Playwright page。
4. 快路径与可靠路径明确区分：可靠路径完整观察、验证、trace；快路径仅允许低风险、确定性动作。

## 6. v0.4 候选需求池

### P0：先达到可发布、可恢复

| ID | 需求 | 用户价值 | 验收标准 |

## 7. 推荐的 v0.4 迭代顺序

### 7.1 进入 v0.4 前的 v0.3 收尾门

以下事项完成前，不建议创建正式 v0.4 发布计划：

1. T01–T10 至少完成第一轮真实网站执行记录。
2. G-04 的敏感信息和提示注入测试通过。
3. 无 Key、浏览器失败、超时、取消和确认拒绝都有可复现记录。
4. 通过一次从零安装验证，确认 README 足以启动项目。

### 7.2 v0.4 M0：状态与契约

- 实现 V4-01、V4-02、V4-05。
- 先用 SQLite 本地实现，不引入云服务。
- 将 RuntimeEvent 扩展为带 task_id、step_id、trace_id 的版本化事件。

### 7.3 v0.4 M1：可靠浏览器执行

- 实现 V4-04、V4-08、V4-11。
- 加入 frame/tab 管理、域名策略和可配置 postcondition。
- 为动态页面与恶意页面建立本地 fixture。

### 7.4 v0.4 M2：性能与观测

- 实现 V4-06、V4-07、V4-09、V4-10。
- 先基准化再优化：以任务成功率、平均端到端时延、模型调用次数、输入 token 和失败率衡量。
- 不以关闭思考或缩短等待作为默认策略；只在可验证的快路径启用。

### 7.5 v0.4 M3：工作台体验

- 实现 V4-12 至 V4-16。
- 将“确认、计划、证据、人工接管”做成主交互，而不是把调试文本堆进时间线。

## 8. 推荐的度量指标

| 指标 | 定义 | 目的 |
| --- | --- | --- |
| 任务完成率 | 通过验证的任务数 / 总任务数 | 衡量真实有效性。 |
| 首次成功率 | 不重试且完成的比例 | 区分“勉强完成”和稳定完成。 |
| 平均/分位时延 | 从提交到终态的 p50/p95 | 发现慢任务。 |
| 每任务模型回合与 token | 规划、决策、提取分别记录 | 控制成本。 |
| 工具验证失败率 | ToolResult.success 为 false 的比例 | 发现定位、等待或页面变化问题。 |
| 人工确认率与拒绝率 | 按风险类别统计 | 校准策略，避免过度打扰或漏拦截。 |
| 恢复成功率 | 从 checkpoint 恢复后完成的比例 | 验证持久化价值。 |
| 安全事件覆盖率 | 攻击/敏感样例通过数 / 总数 | 防止安全承诺停留在文档。 |

## 9. 最终决策

当前建议是：

- **继续把 v0.3 作为“稳定化与验收”版本**，不宣称已发布完成；
- **不在本轮创建正式 docs/v0.4 文件夹**，避免新版本计划掩盖未完成的真实验收和安全工作；
- 当第 7.1 节的收尾门完成后，以本文件第 6 节的 P0/P1 候选需求为基础创建 docs/v0.4，并将需求重新排序、估时和验收。

这不是否定 v0.3：它已具备正确的模块边界、状态机、稳定元素引用、确认门和离线回归。下一阶段的关键不再是增加动作名称，而是把这些能力做成可恢复、可度量、可在真实网站证明的产品。
